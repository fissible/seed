# Schema Wizard ŌĆö Design Spec

**Date:** 2026-03-22
**Scope:** `seed.sh new custom-schema` ŌĆö interactive wizard to generate `.seed` files

---

## Goal

Add `bash seed.sh new custom-schema` ŌĆö an interactive terminal wizard that guides the user through defining a database table schema and writes the result as a `.seed` file consumable by `seed_custom`.

---

## Architecture

```
src/new.sh          ŌåÉ seed_new dispatcher + _seed_new_custom_schema wizard
                      + _seed_new_build_schema pure builder function

seed.sh             ŌåÉ add: source "$SEED_HOME/src/new.sh"

tests/unit/test-new.sh   ŌåÉ unit tests for seed_new and _seed_new_build_schema
```

`seed_new` follows the same pattern as all other generator families: `_seed_cli` calls `seed_${gen}`, so `bash seed.sh new custom-schema` calls `seed_new custom-schema` with zero changes to `_seed_cli` or `seed.sh` beyond adding the source line.

---

## CLI Routing

```bash
seed_new() {
    local sub="${1:-}"; shift || true
    case "$sub" in
        custom-schema) _seed_new_custom_schema "$@" ;;
        *) printf 'Usage: seed.sh new <subcommand>\n' >&2
           printf 'Available: custom-schema\n' >&2
           return 1 ;;
    esac
}
```

`seed_new` with no args or an unknown subcommand exits 1 with usage. No changes to `_seed_cli`.

---

## Wizard Flow

`_seed_new_custom_schema` is the interactive wrapper. It:
1. Guards for TTY
2. Collects input via bash `read` (no external tools)
3. Calls `_seed_new_build_schema` to produce the file content
4. Resolves the output path and writes the file

### Session example

```
$ bash seed.sh new custom-schema

Table name: app_users

Generators:
  1) first_name    2) last_name     3) name
  4) email         5) phone         6) uuid
  7) date          8) number        9) bool
 10) lorem        11) ip           12) url

Column name: id
Select generator [1-12, Enter to finish]: 6

Column name: email
Select generator [1-12, Enter to finish]: 4

Column name: age
Select generator [1-12, Enter to finish]: 8
  Min value [1]: 18
  Max value [100]: 80

Column name: created_at
Select generator [1-12, Enter to finish]: 7
  From date [2000-01-01]: 2020-01-01
  To date [today]:

Fields so far:
  id           uuid
  email        email
  age          number --min 18 --max 80
  created_at   date --from 2020-01-01

Add another field? [Y/n]: n

Save to: tests/fixtures/app_users.seed
Written.
```

### Input rules

- **Table name**: required; empty input re-prompts until non-empty
- **Generator list**: printed once at start, not repeated after each field (the "Fields so far" summary takes that role)
- **Column name**: prompted first each iteration; empty input ŌåÆ done (same exit point as empty generator selection)
- **Generator selection**: number 1ŌĆō12 or Enter (empty) ŌåÆ done; invalid input re-prompts
- **Per-field flags**: prompted only for generators that support them (`number`, `date`, `lorem`); pressing Enter keeps the bracketed default; empty-or-default value omits the flag from the output line
- **SQL type**: inferred from generator (not prompted ŌĆö see inference table below)
- **"Add another field?"**: `Y`/`y`/Enter continues; `N`/`n` ends; also ended by empty generator selection or empty column name

## `_seed_new_build_schema` ŌĆö Pure Builder

Accepts table name and field specs as structured arguments; returns `.seed` file content on stdout. No I/O, no file writing ŌĆö fully testable without stdin mocking.

**Signature (conceptual):**
```bash
_seed_new_build_schema <table> <col1> <gen1> [<col2> <gen2> ...]
```

Each field is a `col gen_spec` pair, where `gen_spec` is the generator name plus any flags (e.g., `number --min 18 --max 80`).

**Output format:**
```
# Generated by seed.sh new custom-schema
# Usage: seed.sh custom --schema <table> --count 10 --format sql
table=<table>
<col1>|<SQL_TYPE>|<gen_spec1>
<col2>|<SQL_TYPE>|<gen_spec2>
...
```

SQL type is inferred per field; the wizard passes `col` and `gen_spec` as parallel arrays to the builder.

---

## SQL Type Inference

| Generator | SQL type |
|---|---|
| `uuid` | `VARCHAR(36)` |
| `number` | `INT` |
| `date` | `TIMESTAMP` |
| `bool` | `BOOLEAN` |
| `ip` | `VARCHAR(15)` |
| `url`, `email` | `VARCHAR(255)` |
| `first_name`, `last_name`, `name`, `phone` | `VARCHAR(100)` |
| `lorem` | `TEXT` |

---

## Output Path Resolution

Resolved by `_seed_new_custom_schema` after the wizard completes:

1. If `tests/fixtures/` exists relative to CWD ŌåÆ write to `tests/fixtures/<table>.seed`
2. If not, and `tests/` exists ŌåÆ prompt `Save to [tests/fixtures/<table>.seed]:` (default shown)
3. If neither exists ŌåÆ prompt `Save to:` with no default (user must supply a path)

If the resolved directory does not exist, create it with `mkdir -p` before writing.

If the target file already exists:
```
tests/fixtures/app_users.seed already exists. Overwrite? [y/N]:
```
Default is No. Declining re-prompts for a new path or exits cleanly.

---

## Error Handling

| Condition | Behaviour |
|---|---|
| Not a TTY (`[ ! -t 0 ]`) | Exit 1: `seed new: requires an interactive terminal` |
| No fields added before "done" | Re-prompt: `No fields defined. Add at least one field.` |
| File already exists, overwrite declined | Re-prompt for a different save path |
| Target directory does not exist | `mkdir -p <dir>` before writing |
| User interrupts with Ctrl-C | Trap `INT`; print newline + `Aborted.`; exit 1 |
| Unknown subcommand to `seed_new` | Exit 1 with usage |

## Testing (`tests/unit/test-new.sh`)

Tests exercise `seed_new` routing and `_seed_new_build_schema` directly. The interactive `_seed_new_custom_schema` is not tested directly (it is a thin I/O wrapper).

- `seed_new` with no args ŌåÆ exits 1, output contains `Usage`
- `seed_new unknown` ŌåÆ exits 1
- `_seed_new_build_schema users id uuid email email` ŌåÆ output contains `table=users`, `id|VARCHAR(36)|uuid`, `email|VARCHAR(255)|email`
- `_seed_new_build_schema orders qty "number --min 1 --max 99"` ŌåÆ output contains `qty|INT|number --min 1 --max 99`
- `_seed_new_build_schema events ts "date --from 2020-01-01"` ŌåÆ output contains `ts|TIMESTAMP|date --from 2020-01-01`; no `--to` flag in output (default/empty omitted)
- SQL type inference: one assertion per generator type verifying the correct SQL type appears in output
- Output always starts with `# Generated by seed.sh new custom-schema` comment header
- Output always contains `# Usage: seed.sh custom --schema ...` comment

---

## Compatibility

- Bash 3.2+: uses `read`, `case`, parameter expansion, indexed arrays (`arr[${#arr[@]}]=val`) ŌĆö no `declare -A`, no `+=`, no `mapfile`
- No external tools in the wizard loop (no `tput`, no `awk`, no `sed`)
- `mkdir -p` is POSIX, accepted dependency
